# Migrating from Phoenix OS v0.23.0 to v0.24.0 webhooks

This guide covers the opt-in durable signed webhook subsystem introduced for
Phoenix OS v0.24.0. It is intended for deployments that already run v0.23.0
jobs, workflows, audit, secrets, Event Bus, plugins, or the Control Plane.

The migration is additive. Existing Event Bus subscribers are not converted,
existing jobs and workflows are not reclassified, and no endpoint, signing key,
subscription, or outbound-network permission is created automatically.

## Compatibility boundary

`RuntimeAssembler` keeps webhooks disabled unless `webhooks_enabled=True` is
supplied. A v0.23.0 assembly can therefore be upgraded without adding webhook
arguments and should preserve its prior service and lifecycle surface.

Do not pass webhook repositories, serializers, egress policies, transport
configuration, signing context, or machine-administration options while
`webhooks_enabled=False`. Phoenix rejects that ambiguous state instead of
silently retaining inactive security-sensitive configuration.

A safe rollout uses conditional keyword arguments:

```python
webhook_options: dict[str, object] = {}

if settings.webhooks_enabled:
    webhook_options = {
        "webhooks_enabled": True,
        "webhook_event_serializers": (
            JobCompletedWebhookSerializer(),
        ),
        "webhook_egress_policies": {
            "public-https": WebhookEgressPolicy("public-https"),
        },
    }

assembler = RuntimeAssembler(
    kernel=kernel,
    events=events,
    capabilities=capabilities,
    configuration=configuration,
    secrets=secrets,
    **webhook_options,
)
```

When the feature flag is false, omit all webhook-specific arguments rather than
passing empty or dormant production configuration.

## Before upgrading

Complete these checks before enabling outbound delivery:

1. Run the v0.23.0 quality gate and record the baseline.
2. Inventory only the Event Bus facts that external systems genuinely require.
3. Define one reviewed serializer for every exported event type.
4. Decide whether webhook state must survive process restart.
5. Provision an exact versioned signing secret in the Phoenix Secrets Manager.
6. Define the smallest egress policy that reaches the intended receiver.
7. Confirm that the receiver verifies the Phoenix signature, timestamp,
   delivery identifier, key version, and attempt number.
8. Choose human administration, service-account administration, or both.
9. Prepare a rollback that disables subscriptions before disabling the runtime.

Do not treat the general Event Bus payload as a webhook body. The serializer is
the allowlisting boundary.

## Step 1: upgrade without enabling webhooks

Upgrade the package or checked-out source while leaving the existing
`RuntimeAssembler` call unchanged. Validate the unchanged assembly first:

```powershell
python -m ruff check .
python -m ruff format --check .
python -m mypy
python -m pytest -q
```

Start and stop the application once with webhooks absent. Verify that no service
whose name begins with `webhooks` appears in the Runtime service map or component
snapshot.

This first deployment establishes that the v0.23.0 behavior remains intact
before any outbound authority is introduced.

## Step 2: define a reviewed event serializer

Each serializer declares exactly one `WebhookEventType` and returns a bounded,
JSON-compatible `WebhookPayload`. Export stable identifiers and allowlisted
business facts only.

```python
from phoenix_os.events import Event
from phoenix_os.webhooks import (
    WebhookEventType,
    WebhookPayload,
)


class JobCompletedWebhookSerializer:
    _event_type = WebhookEventType(
        "jobs.completed",
        resource_filter_fields=frozenset({"job_id"}),
        max_payload_bytes=64 * 1024,
    )

    @property
    def event_type(self) -> WebhookEventType:
        return self._event_type

    def serialize(self, event: Event) -> WebhookPayload:
        job_id = event.payload.get("job_id")
        if not isinstance(job_id, str) or not job_id:
            raise ValueError("jobs.completed requires a non-empty job_id")

        return WebhookPayload(
            event_type=self._event_type,
            data={
                "event_id": str(event.id),
                "job_id": job_id,
                "occurred_at": event.occurred_at.isoformat(),
            },
        )
```

Serializer review should reject:

- credentials, session tokens, API tokens, CSRF proofs, and secret references;
- unrestricted job arguments, workflow inputs or outputs, and plugin metadata;
- local filesystem paths, internal exception text, and raw audit bodies;
- fields that are not required by the receiving contract;
- nondeterministic values that are not facts from the source event.

Changing a serializer schema is a contract change. Prefer a new event type or an
explicitly reviewed schema-version transition rather than silently changing the
meaning of an existing payload.

## Step 3: provision an exact signing-key version

Create signing material through `SecretsManager`. Creation uses an unversioned
`SecretRef`; the returned metadata contains the exact version that must be
stored in the subscription.

```python
import os

from phoenix_os.configuration import SecretValue
from phoenix_os.policy import PrincipalType, SecurityContext
from phoenix_os.secrets import SecretRef, SecretsManager


secrets = SecretsManager()

bootstrap_context = SecurityContext(
    principal="migration:webhook-bootstrap",
    principal_type=PrincipalType.SYSTEM,
    authenticated=True,
    permissions=frozenset({"secret.create"}),
)

created = await secrets.create(
    SecretRef("outbound-hmac", "webhooks"),
    SecretValue(os.urandom(32)),
    bootstrap_context,
)

signing_ref = created.ref
```

Never print, log, serialize, persist in ordinary configuration, or place the
plaintext key in a subscription. Store only `signing_ref`, which includes the
exact immutable version.

The default Runtime signing identity is the authenticated service principal
`phoenix.webhooks`. When the Secrets Manager uses a central `PolicyEngine`, that
policy must allow this principal, or the explicitly supplied
`webhook_signing_context`, to perform the exact secret lease and lease-revocation
operations required for signing.

The reference in-memory secret store is suitable for tests only. A production
deployment is responsible for reviewed encrypted storage and key custody behind
the Phoenix secret interfaces.

## Step 4: define the egress boundary

The default production policy permits public HTTPS destinations on port 443:

```python
from phoenix_os.webhooks import WebhookEgressPolicy


public_https = WebhookEgressPolicy(
    "public-https",
    allowed_ports=frozenset({443}),
    allow_public_networks=True,
    allow_insecure_loopback=False,
)
```

Use a narrower policy when the receiver has stable reviewed address ranges:

```python
partner_https = WebhookEgressPolicy(
    "partner-https",
    allowed_ports=frozenset({443}),
    allowed_networks=("203.0.113.0/24",),
    allow_public_networks=False,
    allow_insecure_loopback=False,
)
```

Phoenix revalidates DNS and destination addresses for every attempt. Do not
weaken this boundary by adding broad private, loopback, link-local, multicast,
or metadata-service ranges. HTTP loopback mode exists only for explicit local
development and must not be carried into production configuration.

## Step 5: choose repository durability

When no repositories are supplied, `RuntimeAssembler` chooses them according to
the configured State Store:

- with a default `StateStore`, Phoenix creates State Store-backed subscription
  and delivery repositories;
- without a State Store, Phoenix creates bounded in-memory repositories;
- explicitly supplied repositories override automatic selection.

In-memory repositories lose subscriptions, delivery history, retry schedules,
and dead-letter state when the process exits. Use them only for tests,
demonstrations, or intentionally ephemeral deployments.

State Store-backed repositories borrow the Runtime-owned State Store. Stopping
webhooks closes repository access but must not be treated as permission to
delete the underlying webhook records.

Capacity defaults are 256 subscriptions and 4096 deliveries. Review capacity,
retention, and the underlying State Store limits before production rollout.

## Step 6: enable the Runtime-owned subsystem

A minimal opt-in assembly requires a `SecretsManager`, at least one serializer,
and at least one named egress policy:

```python
from phoenix_os.configuration import RuntimeAssembler


assembler = RuntimeAssembler(
    kernel=kernel,
    events=events,
    capabilities=capabilities,
    configuration=configuration,
    state=state_store,
    secrets=secrets,
    audit=audit,
    observability=observability,
    webhooks_enabled=True,
    webhook_event_serializers=(
        JobCompletedWebhookSerializer(),
    ),
    webhook_egress_policies={
        public_https.name: public_https,
    },
)

runtime = await assembler.assemble()
await runtime.start()
```

Startup performs the security-sensitive order:

1. create or bind repositories;
2. register reviewed serializers;
3. recover interrupted in-flight deliveries;
4. start the dispatcher worker;
5. subscribe the webhook adapter to the Event Bus.

Shutdown reverses producer ownership: Event Bus selection stops before the
dispatcher and shared repositories are closed last.

The Runtime exposes the composed bundle and individual services under names such
as `webhooks`, `webhooks.manager`, `webhooks.dispatcher`,
`webhooks.recovery`, `webhooks.subscriptions`, and `webhooks.deliveries`.

## Step 7: create the first subscription

Enabling the subsystem creates no subscription. Use the Maintainer Dashboard,
the protected management adapter, or a tightly controlled migration operation.

The following direct manager example is suitable for a reviewed bootstrap or
test harness:

```python
from phoenix_os.policy import PrincipalType, SecurityContext
from phoenix_os.webhooks import (
    WebhookEndpoint,
    WebhookManager,
    WebhookSigningPolicy,
)


manager = runtime.service("webhooks.manager")
assert isinstance(manager, WebhookManager)

maintainer = SecurityContext(
    principal="operator:migration",
    principal_type=PrincipalType.USER,
    authenticated=True,
    permissions=frozenset({"webhook.subscription.create"}),
)

subscription = await manager.create_subscription(
    maintainer,
    name="jobs-completed",
    display_name="Jobs completed",
    event_types=frozenset({"jobs.completed"}),
    endpoint=WebhookEndpoint(
        "https://hooks.example.com/phoenix/jobs"
    ),
    signing=WebhookSigningPolicy(signing_ref),
    egress_policy="public-https",
)
```

The event type must already be registered, the egress-policy name must exist,
the endpoint must pass strict validation, and the signing reference must include
an exact version.

Start with a receiver controlled by the deployment team. Confirm signature
verification and deduplication before adding production consumers.

## Step 8: enable human administration

When the Runtime is assembled in durable operator mode, the webhook manager is
wired into the existing protected Control Plane and Dashboard. Existing
authentication, exact webhook permissions, CSRF verification, no-store
responses, and step-up requirements continue to apply.

Grant only the permissions required by each operator:

- `webhook.health.read`;
- `webhook.subscription.read`;
- `webhook.subscription.create`;
- `webhook.subscription.update`;
- `webhook.subscription.disable`;
- `webhook.subscription.enable`;
- `webhook.subscription.revoke`;
- `webhook.subscription.rotate`;
- `webhook.delivery.read`;
- `webhook.delivery.redrive`.

Do not grant `*` merely to simplify migration.

## Step 9: optionally enable service-account administration

Machine administration remains a separate opt-in:

```python
webhook_service_account_administration_enabled=True
```

It also requires all of the following:

- durable operator mode;
- service accounts enabled;
- a secure fixed-port `ControlPlaneNetworkPolicy`;
- a central `PolicyEngine`;
- replay and audit secrets required by the service-account boundary;
- exact `webhook.*` action scopes and the concrete `webhooks` resource grant.

Do not enable machine administration merely because delivery webhooks are
enabled. Outbound delivery and inbound administrative authority are separate
decisions.

Browser cookies, CSRF proofs, and human step-up proofs are invalid at machine
routes. Machine requests use the existing API-token, nonce, timestamp, replay,
scope, resource, and policy checks.

## Recommended rollout

Use this order for a production migration:

1. Upgrade with webhooks absent and validate unchanged behavior.
2. Deploy reviewed serializers, secret-store support, and conditional feature
   configuration without turning on the production flag.
3. Provision the signing key and record only its exact `SecretRef`.
4. Enable webhooks with an empty subscription repository.
5. Inspect the `webhooks` service and health snapshot.
6. Create one canary subscription to a controlled HTTPS receiver.
7. Emit one allowlisted event and verify the exact signed request.
8. Verify receiver deduplication by delivery identifier.
9. Exercise one retryable response and one terminal response.
10. Confirm delivery history, audit facts, and safe metrics.
11. Expand subscriptions gradually.
12. Enable service-account administration only after separate security review.

## Receiver migration requirements

A receiver should:

- read the raw request body without reserializing it;
- verify `X-Phoenix-Webhook-Signature` with the indicated key version;
- include timestamp, delivery identifier, attempt number, and body digest in the
  expected canonical signature input;
- reject unsupported signature schemes;
- apply an explicit timestamp-age policy;
- compare digests in constant time;
- deduplicate by `X-Phoenix-Webhook-Id`;
- treat retries as the same delivery, not a new business event;
- return a bounded response without relying on redirects.

Keep old signing-key versions available until every delivery and retry using
that version is terminal.

## Health and acceptance checks

After enabling the subsystem, verify:

```python
bundle = runtime.service("webhooks")
manager = runtime.service("webhooks.manager")
dispatcher = runtime.service("webhooks.dispatcher")
```

Operational checks should confirm:

- serializers are registered before the Event Bus adapter reports started;
- no malformed persisted record is accepted;
- no delivery is sent for a disabled, revoked, or missing subscription;
- DNS and destination policy are re-evaluated for every attempt;
- redirect responses are terminal;
- retries remain bounded;
- dead-letter redrive preserves delivery identity and attempt history;
- signing secrets, signatures, canonical bodies, endpoint paths, and internal
  exceptions are absent from administrative responses and logs.

Run the complete repository quality gate after the migration configuration is
added.

## Rollback

Rollback is an administrative sequence, not deletion of state.

1. Disable every active subscription.
2. Wait for currently executing attempts to finish or stop the Runtime through
   its normal lifecycle.
3. Confirm that no new delivery is selected from the Event Bus.
4. Deploy with all webhook arguments omitted and
   `webhooks_enabled` absent or false.
5. Preserve the State Store records and every referenced signing-key version.
6. Investigate and correct the receiver, serializer, secret, policy, or egress
   configuration.
7. Re-enable the same subsystem only after validation.

Do not revoke or delete a signing-key version while a retained delivery can
still reference it. Do not delete webhook repository namespaces as part of a
feature rollback. A rollback to v0.23.0 cannot understand v0.24.0 webhook state,
so retain that state out of band until the deployment returns to v0.24.0 or a
reviewed disposal procedure is executed.

For an emergency network stop, block the receiver at the reviewed egress or
infrastructure boundary and then stop Phoenix normally. Do not modify DNS to a
private or local address in an attempt to redirect traffic; Phoenix is designed
to reject that transition.

## Troubleshooting

### `webhook options require webhooks_enabled`

Webhook-specific arguments were supplied while the feature was disabled. Use
conditional keyword arguments and omit the entire webhook option set.

### `enabled webhooks require a SecretsManager`

Pass the Runtime-owned `SecretsManager` used to resolve exact versioned signing
references.

### `enabled webhooks require at least one event serializer`

Register at least one reviewed `WebhookPayloadSerializer`. Event Bus
subscriptions are not serializers and are never converted automatically.

### `enabled webhooks require at least one egress policy`

Provide a named `WebhookEgressPolicy` and use the same normalized policy name in
each subscription.

### subscription creation rejects the event type

The event type is not registered, or its serializer contract changed. Register
the exact event type before Runtime startup.

### signing fails

Check that the subscription references an existing exact secret version, the
version is active, the signing context is authenticated, and central policy
allows the required lease operations.

### delivery remains retrying or reaches dead letter

Inspect safe attempt categories and receiver status behavior. Do not inspect or
log plaintext signing material. Redrive only after correcting the underlying
cause.

### machine administration is rejected during assembly

Confirm service accounts, operator mode, secure network policy, central policy,
and the explicit webhook machine-administration flag. Delivery webhooks do not
implicitly enable machine routes.

## Post-migration checklist

- [ ] v0.23.0 behavior was validated before enabling webhooks.
- [ ] Every exported event has a reviewed allowlisting serializer.
- [ ] Production state uses a reviewed durable State Store.
- [ ] Signing keys are exact versioned `SecretRef` values.
- [ ] Plaintext signing material is absent from configuration and logs.
- [ ] Egress policies are no broader than required.
- [ ] The receiver verifies signatures and deduplicates deliveries.
- [ ] Canary retry and terminal behavior was exercised.
- [ ] Human permissions are exact.
- [ ] Machine administration was reviewed separately or remains disabled.
- [ ] Rollback preserves state and key versions.
- [ ] The full quality gate passes.
